# Product Update Newsletter Workflow

Use this workflow whenever shipping meaningful product improvements for locksmiths.

## Goals
- Increase feature adoption.
- Reduce support load by pre-answering what changed.
- Keep communication compliant and concise.

## Audience
- Primary: Active locksmith partners.
- Secondary: Inactive locksmiths (reactivation angle).

## Cadence
- Monthly default.
- Ad-hoc for major launches (scheduling, payouts, compliance, dispatch).

## Source of Truth
- Release summary from engineering.
- QA signoff notes.
- Links to relevant portal screens.

## Message Structure
1. Subject line with clear value.
2. 3-5 key improvements.
3. Why this matters in plain language.
4. One CTA per section.
5. Support contact and unsubscribe footer.

## Suggested Subject Patterns
- "New in LockSafe: Better shift scheduling and availability control"
- "LockSafe updates: overnight shifts, 24h options, and smarter dispatch"
- "Product updates for locksmith partners - [Month Year]"

## Release-to-Newsletter Checklist
- [ ] Confirm features are live in production.
- [ ] Confirm screenshots/links reflect current UI.
- [ ] Validate all links in staging.
- [ ] Include UK timezone wording where relevant.
- [ ] Include compliance-safe footer (unsubscribe + support).
- [ ] Send test email to internal seed list.
- [ ] Monitor click/open/error metrics for 48h.

## Distribution Steps
1. Draft from template in `docs/PRODUCT_UPDATE_NEWSLETTER_TEMPLATE.md`.
2. Replace placeholders (month, features, links, CTA).
3. Run internal review (product + support).
4. Send to segment lists in batches.
5. Post same highlights in in-app release notes.

## Metrics to Track
- Open rate.
- Click-through rate by feature block.
- Conversions on key CTA pages.
- Support tickets referencing release confusion.

## Notes
- Keep one canonical newsletter per month, then recycle content into in-app release notes and social snippets.
- Prefer short, practical wording over marketing-heavy language.
